# ADR: plan_harness

## ADR-001: 오케스트레이터 언어는 TypeScript
결정: `tools/harness/` npm workspace로 TS 작성, `tsx`로 실행. Python 예시는 참고만.
이유: 이 리포지토리의 주력 언어가 TS이고 npm workspaces 모노레포 구조가 이미 잡혀있음. 루트 `package.json`에 `"harness"` 스크립트 한 줄만 추가하면 통합 끝. Python을 고르면 별도 런타임·가상환경·린터·포맷터 관리 부담이 생기고, 신규 도구 때문에 저장소 기술 스택이 이원화됨.
트레이드오프: Python 예시 코드를 1:1로 포팅하는 수고가 필요. `subprocess`/`pathlib` 관용구를 Node의 `node:child_process`/`node:fs`로 바꿔야 함. Bash+jq 대비 초기 작성량은 더 많지만 JSON/에러 처리 가독성은 훨씬 좋음.

## ADR-002: 검증은 훅 + 하네스 AC 하이브리드
결정: 기존 `.claude/hooks/` 3종을 **삭제하지 않고 그대로** 유지. 하네스는 step 단위 AC(Acceptance Criteria) 실행과 자가 교정만 담당. 즉 훅=파일 단위 게이트, 하네스=step 단위 게이트로 **레이어 분리**.
이유: 훅은 Edit/Write 호출 시점에 기계적으로 작동하므로 Claude의 자기 보고를 신뢰하는 것보다 강한 보장을 제공(특히 TDD 강제). 예시 프로젝트처럼 훅을 없애면 대화식 Claude Code 세션이 무방비 상태가 되고, 사용자 메모리에 저장된 "자동화 선호 — Stop/PostToolUse 훅으로 Claude가 자율 감지·수정" 방향과도 반대로 감. 훅은 `claude -p` 헤드리스 호출에서도 그대로 작동하므로 하네스 내부 step 실행 중에도 자동 적용됨 → 하이브리드가 자연 성립.
트레이드오프: "검증 포인트가 어디 있는지"가 두 곳으로 분산됨(훅 + 하네스). 디버깅 시 어느 레이어에서 막혔는지 확인해야 함. 대신 훅이 사라지면 생길 TDD 누수·보호 파일 사고 리스크를 피함.

## ADR-003: 상태 파일을 phases/index.json으로 단일화
결정: 신규 플랜부터 `progress.txt` / `feature_list.json` 생성 중단. `docs/plan_<name>/phases/index.json`이 모든 step 상태의 단일 소스. 전역 현황은 `docs/plans-index.json` 한 파일로.
이유: `progress.txt`(자유 서술)과 `feature_list.json`(boolean 체크리스트)은 상태 필드(status/summary/timestamps)까지 합치면 `phases/index.json` 스키마와 **거의 완전히 중복**. 세 파일 병존 시 갱신 누락이 매 세션 발생했고(지난 plan_tdd에서 feature_list.json을 내가 놓친 사고), 자동화 훅이 세 파일을 동기화하려면 복잡도가 급증함. 하네스가 JSON 하나만 읽고 쓰면 구현이 단순해지고 stale 가능성도 제거됨.
트레이드오프: 사람이 눈으로 훑기 어려움(자유 서술 vs JSON). "지금 어디까지 됐지?"를 묻는 인간용 서사는 없어지고, JSON을 훑어야 함. 필요해지면 `scripts/plan-status.ts` 같은 CLI로 예쁘게 렌더링하는 건 나중에 추가 가능(지금은 YAGNI).

## ADR-004: 설계=대화식, 실행=헤드리스로 분리
결정: phase 설계(PRD/ARCH/ADR/phases 디렉터리 생성)는 사용자와 대화식 Claude Code 세션에서 수행(`/harness` 슬래시 커맨드). 설계된 step 큐의 **실행**만 `execute.ts`가 `claude -p`로 돌림.
이유: 기능 설계는 본질적으로 사용자 판단·트레이드오프·아키텍처 논의가 섞인 양방향 작업이라 자동화에 맞지 않음. 반면 설계가 끝난 다음의 step 실행은 "지시서를 따라 코드 쓰기 + AC 확인"이라 자율 실행에 적합. 두 단계를 섞으면 하네스가 설계 결정까지 해버려 사용자 의도를 벗어날 위험이 큼. 예시 Python 스크립트도 명시적으로 "D. 파일 생성" 이후에 "E. 실행"이 오도록 워크플로우를 분리.
트레이드오프: 설계 대화 중엔 훅 게이팅만 작동하고 하네스의 AC 재실행은 없음. 설계 단계에서 코드를 건드릴 일 자체가 적으므로 실무적 손실은 작음.

## ADR-005: Trust but Verify — 하네스가 AC를 재실행
결정: Claude 세션이 `status=completed`로 보고해도 `execute.ts`가 해당 step의 AC 커맨드를 **한 번 더 실행**해 실제 통과 여부를 확인. 불일치면 자동 error 처리하고 재시도 루프 진입.
이유: 예시 Python 스크립트는 Claude의 자기 보고를 100% 신뢰함. 모델이 환각·게으름·이해 오류로 "됐다"고 보고해도 감지 못함. 헤드리스 자율 실행은 사용자 확인이 없으므로 실제 검증의 마지막 방어선이 필요. AC가 "실행 가능한 커맨드"인 한(PRD의 설계 원칙) 재실행 비용은 저렴함.
트레이드오프: AC 커맨드를 2번 돌리므로 step당 실행 시간이 증가함(특히 `npm run build`가 포함되면 수 분 추가). 그러나 이중 실행으로 커밋 시점의 품질을 기계적으로 보장하는 가치가 더 큼. 느리면 AC를 경량 커맨드(lint+type-check)로 짜면 됨.

## ADR-006: docs/plan_<name>/phases/ — 설계·실행·현황을 한 폴더에
결정: 플랜별로 설계 문서(PRD/ARCH/ADR)와 실행 큐(phases/)를 동일한 `docs/plan_<name>/` 하위에 둠. 루트에 `phases/` 폴더를 따로 만들지 않음(예시 프로젝트와 다름).
이유: 사용자가 "설계와 실행, 실행 현황을 plan 단위로 넣어놓고 싶다"고 명시. 동일 플랜의 문서가 물리적으로 한 곳에 모여 있으면 검토·삭제·아카이브가 플랜 단위로 깔끔히 움직임. 예시 프로젝트의 `phases/<name>/` 루트 배치는 설계 문서가 별도 관리라는 전제였음. 이 프로젝트는 `docs/plan_*/` 관례가 먼저 자리잡았으므로 그 아래로 편입하는 게 일관적.
트레이드오프: execute.ts 경로 해석이 약간 복잡해짐(예시의 `phases/<dir>/` → `docs/plan_<name>/phases/`). 대신 상대 경로가 플랜과 결합돼 있어 파일 이동/삭제 시 실수 위험이 낮음.

## ADR-007: YAGNI로 실행 기능 축소
결정: MVP에서 resume/watch, 병렬 실행, step 롤백, 하네스 자체의 훅 on/off 플래그, CI 자동 트리거, 타임아웃 커스터마이즈를 모두 제외.
이유: 각 기능의 트리거 조건이 아직 실측되지 않음. 순차 실행 + 실패 시 수동 복구로 초기 사용 경험이 충분한지 먼저 확인해야 함. 예시 Python 스크립트도 이 중 대부분을 제공하지 않고 프로덕션에서 쓰이고 있음. 미리 넣으면 유지·테스트·문서 비용만 늘고 실제 사용 패턴과 어긋난 설계로 굳을 위험.
트레이드오프: 긴 phase 중간에 장애 복구 시 사용자가 JSON을 직접 편집해야 함. 이 마찰이 실제로 문제가 되는 시점에 각 기능을 개별 ADR로 추가하는 절차가 바람직.
