1. 전통적인 웹 개발 환경에서 프론트엔드는 이미 구축된 백엔드 서버의 API URL로 요청을 보내 데이터를 주고받는 역할을 수행해 왔다.
2. 이 과정에서 프론트엔드 개발자는 외부 API 엔드포인트를 호출하고 응답을 받아 UI에 렌더링하는 클라이언트 중심의 로직에만 집중하게 되었다.
3. 하지만 클라이언트에서 직접 외부 API를 호출하는 방식에는 몇 가지 한계가 존재한다.
   - API 호출에 필요한 보안 키가 브라우저에 노출되어 보안 취약점이 발생할 수 있다.
   - 타 도메인 API 사용 시 발생하는 CORS 정책은 브라우저 보안 규정이므로, 클라이언트 요청만으로는 통제가 불가능하다. 이를 서버 환경인 Route Handlers에서 호출하면 브라우저의 CORS 제약을 우회할 수 있다.
   - 복잡한 데이터를 가공하거나 여러 API를 조합해야 할 때 이를 클라이언트에서 처리하면 브라우저의 연산 부담이 커지고 로직이 복잡해지는 경우가 있다.
4. 이러한 문제를 해결하고 프론트엔드 환경 내에서 자체적인 서버 로직을 구현하기 위해 `app` 디렉토리 내에서 웹 표준 API를 기반으로 서버 사이드 엔드포인트를 정의할 수 있는 Route Handlers를 제공한다.
5. 이는 Next.js 앱 내부의 `app` 디렉토리 내에 위치하며, 특정 경로에 대해 웹 표준 `Request` 및 `Response` API를 사용하여 사용자 정의 요청 핸들러를 만들 수 있게 해주는 서버 사이드 API 엔드포인트이다.
6. `route.ts`를 활용하면 프론트엔드 프로젝트 내부에 '미니 백엔드' 역할을 하는 계층을 구성할 수 있다. 클라이언트(브라우저)가 외부 API를 직접 호출하는 대신, 우리 프로젝트 내부의 `api/example/route.ts` 경로로 요청을 보낸다. 그러면 서버 환경에서 동작하는 `route.ts` 파일이 클라이언트 대신 외부 API와 통신하거나 데이터베이스에 접근하여 필요한 처리를 수행한다. 이 과정에서 보안 키는 서버 환경 변수로 안전하게 관리되며, 브라우저가 아닌 서버 간 통신이므로 CORS 제약에서도 자유로워진다.
7. 이 과정에서 보안 키는 서버 환경 변수로 안전하게 관리되며, 브라우저가 아닌 서버 간 통신이므로 CORS 제약에서도 자유로워진다.
